# 1. WHAT IS DRIFT TRIAGE CO-PILOT?

## Simple Definition

Imagine you have a **robot that makes decisions** (like approving loans or predicting if someone will buy a product). 

Over time, the world changes. People change. The economy changes. So your robot gets **worse at its job** because it learned from old data.

**"Drift"** = when the robot's inputs start looking different from what it trained on.

**"Triage"** = like in a hospital emergency room - deciding which patient to treat first, and how.

**"Co-Pilot"** = an AI assistant that helps manage this problem automatically.

## What It Does (The Mission Statement)

Train a binary classifier on the UCI Bank Marketing dataset. Register it. Serve it. Watch it. Build a **LangGraph supervisor** that consumes the drift signal, decides what to do, dispatches slow consequences (replay the test set, retrain, rollback) through a Redis-backed queue, persists its state across restarts, and pauses for human approval before any change to Production reaches the registry.

## The Two Required Tracks

| Track | Focus | Key Components |
|-------|-------|----------------|
| **MLOps** | Artifacts, validation, observability | PSI, chi², end-to-end serving, promotion gates |
| **Agentic Systems** | Multi-agent topologies, memory, persistence | LangGraph, checkpoints, long-running tools |

**Both are required. Neither is optional.**

## Critical Requirements

- This is **not** a Colab notebook. It is a real repository that comes up with `docker-compose up` from a clean clone.
- The platform and the agent each survive the other crashing.
- The agent's decision logic is regression-tested in CI.
- The queue is idempotent.
- None of these are stretch goals.

## What is a Term Deposit?

A **term deposit** = A savings account where you lock money for a fixed time (6 months, 1 year, etc.) for higher interest.

**The bank's problem:**
- Calling people costs money
- If you call someone who will NEVER buy → wasted money
- If you call someone who WILL buy → profit!

**Your model's job:** Predict YES (1) or NO (0) for each customer.

## KNOWN TRAPS - CRITICAL!

| Trap | Problem | Solution |
|------|---------|----------|
| **duration** | This feature is recorded AFTER the call ends. It leaks the target (long calls = more likely to buy). In production, you don't know duration before calling! | **DROP this column entirely** |
| **pdays == 999** | 999 means "client was never contacted before" (sentinel value) | Bin it OR create a flag column: `was_contacted_before = (pdays != 999)` |
| **'unknown' values** | Categories marked 'unknown' are NOT missing data | Treat as a real category. It tells you: "the bank doesn't know this info about the customer" (which is informative!) |

## Features to Know

**Numerical features (PSI monitors these):**
- `age` (21 to 90 years old)
- `euribor3m` (interest rate: 0.5% to 5%)
- `cons.price.idx` (consumer price index)
- `campaign` (number of calls made to this client)
- `previous` (number of previous contacts)
- `emp.var.rate` (employment variation rate)

**Categorical features (Chi² monitors these):**
- `job` (admin, technician, services, retired, student, etc.)
- `marital` (married, single, divorced)
- `education` (basic, high school, university)
- `contact` (telephone, cellular)
- `month` (jan, feb, ..., dec)
- `day_of_week` (mon, tue, ...)

## Decision #55: Output-Distribution Drift (COMPLETE — NEW)

**The brief explicitly requires:** "output-distribution drift" in addition to PSI and chi².

| Aspect | Our Decision |
|--------|--------------|
| **Choice** | ✅ **PSI on binned predicted probabilities (10 equal-width bins, 0.0 to 1.0)** |
| **Why** | The model's output distribution (the `predict_proba` score for class 1) must be stable over time. PSI on 10 bins of the probability score directly measures this. Same threshold scale as numeric PSI. This is the fastest signal that the model is behaving differently even before feature drift surfaces — a canary in the coal mine. |

**Implementation:**
```python
def compute_output_drift(reference_probs: list[float], current_probs: list[float], bins: int = 10) -> float:
    edges = np.linspace(0.0, 1.0, bins + 1)
    ref_hist, _ = np.histogram(reference_probs, bins=edges)
    cur_hist, _ = np.histogram(current_probs, bins=edges)
    return compute_psi(ref_hist, cur_hist)
```

**Severity thresholds:** Same PSI scale as Decision #6. No economic feature escalation applied to output drift.

| Status | ✅ COMPLETE |

## Decision #56: Retrain Worker — Exact Responsibilities (COMPLETE — NEW)

The retrain worker picks up a `RETRAIN` or `RETRAIN_URGENT` task from the Redis queue and executes these steps in order:

1. **Load data:** Pull training reference data + last 7 days of `predictions_log` rows from Postgres (recent inference distribution context)
2. **Retrain pipeline:** Run the full sklearn `Pipeline` (preprocessor + LightGBM) with `random_state=42`
3. **Tune threshold:** On the validation split, find the highest threshold where `recall >= 0.75`
4. **Run promotion gate checks:** AUC > 0.80, Recall >= 0.75 at tuned threshold, `duration` absent from features, `pdays` handled as flag, drift acceptable on reference data, model card complete
5. **If gate passes:** Register new version in MLflow with artifact triple (model binary, input schema, model card with hash + env fingerprint). Set alias to `Staging` only — NEVER Production
6. **Write result to Postgres:** Update task row with `status=COMPLETED`, `new_model_version=N`
7. **If gate fails:** Update task with `status=FAILED`, reason logged. Move to DLQ. Alert dashboard
8. **Agent resumes on poll:** Reads task status. Creates a NEW HIL request for Production promotion of the Staging version
9. **Human approves HIL → Agent calls `/promote`** → model promoted to Production

**Critical rule:** The worker NEVER promotes to Production directly. It only creates a Staging version. Production promotion is always agent + HIL.

**Implementation:**

```python
def triage(psi, chi2_pvalue, feature):
    psi_band  = psi_threshold(psi)          # LOW/MED/HIGH/CRIT
    chi2_band = chi2_threshold(chi2_pvalue) # LOW/MED/HIGH or None
    severity  = max(psi_band, chi2_band) if chi2_band else psi_band  # worst signal
    if feature in ECONOMIC_FEATURES and psi > 0.15:
        severity = escalate(severity)       # +1 level
    return severity
```

```python
def validate_sync(checkpoint, mlflow_registry):
    issues = []
```

```python
def generate_idempotency_key(investigation, action_type):
    components = [
        action_type,                    # "retrain"
        investigation.drift_feature,    # "euribor3m"

# Queue checks before adding
```

# 9. QUEUE & TASKS DECISIONS (COMPLETE)

## Decision #29: What goes into the idempotency key? (COMPLETE)

**Our Decision:** `hash(action_type + feature + date_hour + severity)`

```python
def generate_idempotency_key(investigation, action_type):
    components = [
        action_type,
        investigation.drift_feature,
        investigation.drift_date_hour,
        investigation.severity
    ]
    key = "_".join(components)
    return hashlib.md5(key.encode()).hexdigest()[:16]
```

**Why:** Same drift on same feature in same hour → same key → duplicate ignored. Different hour → different key → new retrain allowed.

```python
    def __init__(self):

        }
        self.call_count = 0
        self.last_prompt = None
    
    def generate(self, prompt):
        self.call_count += 1
        self.last_prompt = prompt
        for keyword, response in self.responses.items():
            if keyword.lower() in prompt.lower():
                return response
        return self.responses["default"]
    
    def reset(self):
        self.call_count = 0
        self.last_prompt = None
```

**Why:** Not too simple (has keyword matching), not over-engineered (no fancy patterns). Maintainable, testable, realistic.

```python
def call_llm_with_fallback(prompt):
    # Try OpenAI with retries
    for attempt in range(3):
        try:
            return openai_client.generate(prompt)
        except Exception as e:
            wait = 2 ** attempt
            time.sleep(wait)
    
    # Fallback to Groq
    try:
        return groq_client.generate(prompt)

    # Final fallback: template text
```

# 21. COMPLETE EXPLANATION - HOW EVERYTHING WORKS TOGETHER

## What We Built and Why

We have designed a complete **Drift Triage Co-Pilot** system that automatically monitors a machine learning model in production, detects when the data starts changing (drift), and takes intelligent action to keep the model accurate - all while keeping a human in the loop for important decisions.

## The Core Problem We're Solving

Banks use ML models to predict which customers will buy term deposits. But the economy changes. Interest rates (euribor3m) go up and down. Customer behavior changes. A model trained on old data becomes less accurate over time. This is called **drift**.

## Our Solution Architecture

The system has four main layers:

### Layer 1: The Platform (MLflow + FastAPI) - PENDING

This is where the ML model lives. It:
- Trains a binary classifier on bank data
- Registers models in MLflow with versioning
- Serves predictions via a FastAPI endpoint
- Continuously calculates drift using PSI (for numbers) and Chi² (for categories)
- Sends webhook alerts when drift is detected

*These decisions are pending and will be made during implementation.*

### Layer 2: The Agent (LangGraph Supervisor) - COMPLETE

This is the brain of the system. We made 46 decisions about how it works:

- **Webhooks, not polling** - We chose webhooks for real-time drift detection. Economic features like euribor3m change fast; we can't afford to wait 60 seconds for a poll.
- **Three sub-agents with a supervisor** - Triage (decision tree, NO LLM) determines severity from PSI numbers. Action (rules for 90%, LLM for edge cases) decides what to do. Comms (pure LLM) talks to humans.
- **Triage uses a decision tree, not an LLM** - PSI is a number. Comparing it to 0.2 doesn't need an AI. This saves money and latency.
- **Action uses rules for 90% of cases, LLM only for edge cases** - Recent retrain? Multiple drifts? Borderline scores? That's where the LLM adds value. Standard cases use fast, deterministic rules.
- **Comms uses an LLM for all human-facing text** - Humans don't read PSI numbers. They read sentences. The LLM translates "PSI=0.35, severity=HIGH" into "Interest rates have risen significantly. The model needs retraining."

### Layer 3: Persistence & Checkpoints - COMPLETE

The agent saves its brain state to Postgres at 7 milestones and before any risky operation. If the agent crashes, it resumes exactly where it left off. We solved the 5 "Think About" problems:

1. **Checkpoint & Registry Sync** - MLflow is source of truth. Agent auto-updates to match.
2. **Missing Model URI** - Ask human via HIL. No automatic fallback (safety first).
3. **Duplicate Retrains** - Idempotency keys (hash of action+feature+hour+severity) with TTL (24h for retrain, 1h for others).
4. **Stale HIL Approval** - 10 minute expiry. New drift adds warning, doesn't auto-cancel.
5. **Direct Promotion Bypass** - Agent + emergency key. Normal = agent+HIL, emergency = secret header.

### Layer 4: Async Tasks (Redis Queue + DLQ) - COMPLETE

Slow actions (retraining, replaying test sets) go into a Redis queue. Worker picks them up. If a task fails, it retries with exponential backoff (1,2,4,8,16 seconds). After 5 failures, it goes to the Dead Letter Queue for human review. Idempotency keys prevent duplicate tasks.

### Layer 5: Dashboard (Streamlit) - COMPLETE

The human window into the system. Shows registry state, active investigations, queue depth, and HIL inbox with Approve/Reject buttons. Updates in real-time via WebSocket (with polling fallback). Refresh every 2 seconds.

## Key Design Philosophies

1. **LLM only where it adds value** - Severity detection uses decision trees (free, fast). Action decisions use rules (90% of cases). Edge cases use LLM. Comms agent uses LLM for human text.
2. **Safety first** - No automatic production changes without HIL approval.
3. **Resilience** - Checkpoints survive crashes. Queue persists. Retries with backoff.
4. **Testability** - MockLLM with keyword-based responses. Full trajectory snapshots. 1e-12 fidelity test.
5. **Professional Python** - Async everywhere. Pydantic at boundaries. Dependency injection. Structured logging. No antipatterns.

## The Complete Flow

1. Customer calls prediction endpoint → model predicts
2. Drift calculator compares recent data to training data
3. PSI > 0.2 detected on euribor3m → drift alert
4. Webhook sent to LangGraph Supervisor
5. Supervisor creates investigation, saves checkpoint
6. Triage agent (decision tree) → severity = HIGH (escalated to CRITICAL because economic feature)
7. Action agent (rules) → RETRAIN_URGENT
8. Supervisor pauses, sends HIL request to dashboard
9. Human clicks APPROVE (within 10 minutes)
10. Supervisor validates (no new drift, not expired) → dispatches to Redis queue
11. Worker picks up task, retrains model with recent data
12. New model v3 saved to MLflow
13. Promotion gate checks (AUC>0.80, Recall>=0.75, etc.) → passes
14. v3 promoted to production
15. Dashboard updates: "Production model: v3 (AUC 0.85)"

## Decision #58: Bootstrap / Initial Training Flow (COMPLETE — NEW)

**The problem:** On a fresh `docker-compose up` the model does not exist yet. The platform cannot serve predictions, the drift calculator has no reference distribution, and MLflow is empty. Something must train and register the model before the system is usable.

**Our Decision:** ✅ **A dedicated `trainer` Docker service that runs once and exits.**

- `trainer` is a separate Docker service with `restart: "no"` so it runs exactly once per `docker-compose up`
- It trains the LightGBM pipeline, tunes the threshold, registers `v1` in MLflow with the artifact triple, computes and stores reference distributions in Postgres, then exits with code 0
- All other services that need the model (`platform`, `agent`, `worker`) have `depends_on: trainer: condition: service_completed_successfully`
- On subsequent restarts (crash recovery), `trainer` does NOT re-run because `restart: "no"` — the model already exists in MLflow
- If MLflow already has a registered Production model on startup, `trainer` skips retraining (idempotent check: `if mlflow_client.get_latest_versions("bank-classifier", stages=["Production"]): sys.exit(0)`)

**Why not a startup event inside FastAPI?** The platform would have to block on training (minutes) before accepting requests. A separate service keeps concerns separated and the health check chain clean.

## Decision #61: Worker Service Design (COMPLETE — NEW)

**Our Decision:** ✅ **Custom async polling loop — no Celery, no arq.**

**Why:** Celery adds broker configuration complexity and arq enforces a specific task format. A custom loop is ~60 lines, fully transparent, and both teammates can explain every line on Friday (project rule: no vibe coding).

**How it works:**
- The worker runs a `while True` async loop
- Uses Redis `BLPOP` with a 5-second timeout on the main queue key — blocks until a task arrives or timeout
- On receiving a task: deserializes JSON, checks idempotency key, executes the appropriate handler (retrain / replay / rollback)
- On success: updates task row in Postgres to `COMPLETED`, releases idempotency key
- On failure: increments retry count, re-queues with exponential backoff delay using Redis `ZADD` on a delayed queue; after 5 failures moves to DLQ list
- Worker polls the delayed queue every second and moves ready tasks back to the main queue

**Delayed queue pattern:** Redis sorted set where the score is the Unix timestamp when the task becomes eligible. Worker checks `ZRANGEBYSCORE delayed_queue 0 {now}` and moves matching tasks to main queue.
