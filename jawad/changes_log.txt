COMPLETE WORKFLOW — AGENT + QUEUE + WORKER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STEP 1 — Platform detects drift, sends webhook
  POST /webhook   body: {feature_name, psi_score, chi2_pvalue, model_version, ...}
  Agent returns 202 immediately. Creates DriftInvestigation row (status=RUNNING).
  Fires graph as asyncio background task.

STEP 2 — LangGraph graph executes: START → supervisor → triage → supervisor → action → supervisor → comms → END

  SUPERVISOR (pure router, no LLM):
    - Reads state["next_node"], calls Command(goto=next_node)
    - MAX_STEPS=10 guard (loops back to supervisor after each node)
    - Never makes decisions — only routes

  TRIAGE NODE (pure decision tree, no LLM):
    Bands by PSI:
      PSI < 0.10  → LOW
      PSI 0.10-0.20 → MED
      PSI 0.20-0.25 → HIGH
      PSI > 0.25  → CRIT
    Chi² override: if chi2_pvalue < 0.01 and severity == LOW → elevate to MED
    Economic escalation: if feature in (euribor3m, cons.price.idx) and PSI > 0.15 → +1 level
    Writes severity to DB (drift_investigations.severity).
    Returns next_node="action"

  ACTION NODE (7-rule chain + LLM for ~10% edge cases):
    Rule priority (first match wins):
      1. model_uri_missing            → SWITCH_TO_FALLBACK
      2. model_auc < 0.65             → ROLLBACK
      3. severity == CRIT             → ROLLBACK
      4. economic + HIGH              → RETRAIN_URGENT
      5. economic + MED               → RETRAIN_SCHEDULED
      6. recent_retrain + HIGH/MED    → REPLAY_TEST_SET
      7. HIGH → RETRAIN_SCHEDULED  |  MED → REPLAY_TEST_SET  |  LOW → MONITOR

    Idempotency: SHA-256 of (action + feature + UTC_hour + severity)
      - If key exists in DB → return [DUPLICATE], skip HIL, skip queue push
      - TTL: 24h for RETRAIN, 1h for others

    HIL actions (ROLLBACK, RETRAIN_URGENT, RETRAIN_SCHEDULED, SWITCH_TO_FALLBACK):
      - Creates HILApproval row (status=PENDING, expires in 10 min)
      - Sets investigation status=AWAITING_HIL
      - Calls interrupt() → graph pauses, waits for operator

    Non-HIL actions (REPLAY_TEST_SET, MONITOR):
      - Dispatches task to Redis immediately (LPUSH queue:tasks)

    Resume detection: on graph re-execution after approve/reject,
      checks DB for APPROVED or REJECTED HILApproval for this investigation_id
      → sets is_resume=True, skips re-creating HIL and idempotency write

  COMMS NODE (pure LLM — always runs):
    - Builds prompt with feature, PSI, severity, action, HIL status
    - Calls gpt-4o-mini → Groq fallback → template string fallback
    - Writes comms_message + status=COMPLETED to DB
    - Duplicate path (action_rationale starts with [DUPLICATE]): skips LLM,
      writes a standard dedup message + status=COMPLETED to DB

STEP 3 — HIL flow (for actions requiring approval)
  Dashboard / operator polls GET /approvals → sees pending HIL

  Approve:
    POST /investigations/{id}/approve  body: {"note": "..."}
    → HILApproval.status = APPROVED
    → investigation.status = RUNNING
    → graph.ainvoke(Command(resume={hil_approved: True, ...})) as background task
    → action_node resumes after interrupt(), dispatches task to Redis
    → comms_node writes final message, status=COMPLETED

  Reject:
    POST /investigations/{id}/reject   body: {"note": "..."}
    → HILApproval.status = REJECTED
    → graph resumes, action_node sees hil_approved=False, skips queue push
    → comms_node writes rejection message, status=COMPLETED

STEP 4 — Worker consumes tasks from Redis
  Worker runs async BLPOP loop on queue:tasks (timeout=5s)
  HANDLERS dict routes task_type to handler:
    RETRAIN_SCHEDULED / RETRAIN_URGENT → handle_retrain()
    ROLLBACK / SWITCH_TO_FALLBACK      → handle_rollback()
    REPLAY_TEST_SET                    → handle_replay_test()

  Retry: 5 attempts, exponential backoff 1s/2s/4s/8s/16s
  After 5 failures: push to queue:tasks:dlq (Dead Letter Queue)

  handle_retrain():
    1. POST {platform_url}/retrain           → platform trains new model
    2. POST {agent_url}/investigations/{id}/notify_retrain_complete
       body: {model_name, model_version}
       → agent creates PROMOTE_TO_PRODUCTION HILApproval

  handle_rollback():
    POST {platform_url}/registry/promote    candidate_version=previous
    (also used for SWITCH_TO_FALLBACK)

  handle_replay_test():
    POST {platform_url}/replay-test         → platform re-scores test set, returns metrics

STEP 5 — 2nd HIL loop (retrain path only)
  Operator sees PROMOTE_TO_PRODUCTION HIL in dashboard
  POST /investigations/{id}/approve
    → _resolve_hil detects proposed_action == PROMOTE_TO_PRODUCTION
    → calls POST {platform_url}/registry/promote  (platform_promote_failed logged if platform down — non-fatal)
    → investigation.status = COMPLETED
    → NO LangGraph resume (graph already ended after first HIL)
