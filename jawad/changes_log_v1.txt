[backend/agent/settings.py]
  CHANGE: extra="forbid" → extra="ignore" in Settings model_config
  REASON: A8 — .env is shared with platform/MLflow; Settings crashed with
          "6 extra inputs not permitted" for keys it didn't own.

[backend/agent/agents/checkpointer.py]
  CHANGE: Rewrote from bare async function (await AsyncPostgresSaver.from_conn_string())
          to @asynccontextmanager yielding inside `async with AsyncPostgresSaver...`
  REASON: langgraph-checkpoint-postgres newer API uses async context manager pattern;
          calling `await` on it returned a GeneratorContextManager, not a saver,
          causing: TypeError: _AsyncGeneratorContextManager can't be used in 'await'

[backend/agent/main.py]
  CHANGE: Import updated: `build_checkpointer` → `checkpointer_lifespan`
  CHANGE: Lifespan restructured to `async with checkpointer_lifespan(settings) as saver:`
          with yield nested inside the context manager block
  REASON: Required by the checkpointer.py rewrite above; yield must stay inside the
          async context manager or the Postgres connection closes before the app shuts down.

[backend/agent/core/logging.py]
  CHANGE: Removed `structlog.stdlib.add_logger_name` from processors list
  REASON: PrintLogger (structlog's default in non-stdlib mode) has no `.name` attribute;
          this caused AttributeError: 'PrintLogger' has no attribute 'name' on startup.

[backend/agent/db/models.py]
  CHANGE: Full rewrite of all DateTime column definitions — introduced
          TZ = DateTime(timezone=True) shorthand; applied to every Mapped[datetime] column
  REASON: Mixed timezone-naive/aware datetimes caused:
          DBAPIError: can't subtract offset-naive and offset-aware datetimes
          Also fixed a secondary bug: ArgumentError: May not pass type_ positionally
          and as a keyword (duplicate type spec in original column definitions).
  NOTE: Required DROP + recreate of all agent tables because SQLAlchemy can't
        ALTER existing columns from timezone-naive to timezone-aware in-place.

[backend/agent/agents/nodes/triage.py]
  CHANGE: Added `config: RunnableConfig` parameter to triage_node signature
  CHANGE: Added DB write after triage computation — loads DriftInvestigation by
          investigation_id from state, sets inv.severity = severity, commits
  CHANGE: Added imports: uuid, RunnableConfig, DriftInvestigation
  REASON: A4 — severity was stuck as "PENDING" in the DB; node computed it correctly
          but never persisted it.

[backend/agent/agents/nodes/comms.py]
  CHANGE: Added `config: RunnableConfig` parameter to comms_node signature
  CHANGE: Added DB write after LLM generates message — sets inv.comms_message = message
          and inv.status = InvestigationStatus.COMPLETED, then commits
  CHANGE: Added imports: uuid, RunnableConfig, DriftInvestigation, InvestigationStatus
  REASON: A3 — comms_message was never written to DB; status never reached "completed".

[backend/agent/schemas/hil.py]
  CHANGE: Removed `approved: bool` field from HILApprovalRequest
  REASON: Field was redundant — /approve and /reject are separate endpoints that
          already encode the decision in the URL. The field caused a 422 on every
          approve call because callers naturally omit it.

[backend/agent/routers/investigations.py]
  CHANGE: Added `from sqlalchemy.orm import selectinload`
  CHANGE: Replaced `session.get(DriftInvestigation, ...)` with
          `session.scalar(select(...).options(selectinload(hil_approvals)))` in
          get_investigation, _resolve_hil (entry), and _resolve_hil (return)
  REASON: SQLAlchemy async forbids lazy-loading relationships; accessing
          row.hil_approvals after a bare session.get() raised MissingGreenlet 500.

[backend/agent/agents/nodes/action.py]
  CHANGE: Added DB query for APPROVED HIL at start of action_node to detect resume:
          `is_resume = approved_hil is not None` where approved_hil is fetched via
          select(HILApproval).where(investigation_id=..., status=APPROVED)
  CHANGE: Idempotency check + write skipped when is_resume=True
  CHANGE: HIL record creation + AWAITING_HIL status write skipped when is_resume=True
  CHANGE: hil_approval_id read from the approved_hil DB row on resume
  REASON: LangGraph re-executes the entire node on resume (interrupt() reruns from
          top). hil_approval_id is a local variable never returned to state on first
          execution (interrupt raises before return), so state.get("hil_approval_id")
          is always None on resume — the only reliable signal is the APPROVED HIL
          record committed to DB by _resolve_hil before the resume task is created.

  CHANGE: inv.proposed_action = action written to DB in the HIL path (inside the
          existing session block) and in the non-HIL path (new session block)
  REASON: proposed_action was computed and stored in LangGraph state but never
          persisted to the drift_investigations row — GET /investigations always
          returned proposed_action=null.

[docker-compose.yml]
  CHANGE: Postgres host port changed from "5432:5432" → "5433:5432"
  REASON: Local Odoo installation owns port 5432; Docker Postgres couldn't bind,
          causing connection refused on agent startup.

  CHANGE: handle_retrain() — POST {platform_url}/retrain (step 1);

  REASON: Retrain is the critical path: platform trains new model → agent
          creates PROMOTE_TO_PRODUCTION HIL for human sign-off.

[backend/worker/tasks/rollback.py]  (new)
  CHANGE: handle_rollback() — POST {platform_url}/registry/promote with
          candidate_version=previous + reason. Handles ROLLBACK and
          SWITCH_TO_FALLBACK task types.
  REASON: Both actions mean "go back to last known good" — same platform call.

[backend/worker/tasks/replay_test.py]  (new)
  CHANGE: handle_replay_test() — POST {platform_url}/replay-test with
          investigation_id + feature_name; logs returned metrics.
  REASON: REPLAY_TEST_SET re-scores held-out test set to verify drift is real.

[backend/agent/schemas/hil.py]
  CHANGE: Added RetrainCompleteNotification(model_name, model_version)
  REASON: Schema for the worker's notify_retrain_complete call to the agent.

[backend/agent/routers/investigations.py]
  CHANGE: Added imports: timedelta, httpx, RetrainCompleteNotification, get_settings
  CHANGE: _resolve_hil — added PROMOTE_TO_PRODUCTION branch:
          if HIL action == PROMOTE_TO_PRODUCTION and approved → call
          platform /registry/promote using model info stored in drift_payload;
          platform failure is logged as warning (non-fatal, graph already done);
          investigation → COMPLETED. Original LangGraph resume path unchanged.
  CHANGE: Added POST /{id}/notify_retrain_complete endpoint — creates
          PROMOTE_TO_PRODUCTION HILApproval, stores pending_model in
          drift_payload JSONB, sets status=AWAITING_HIL.
  REASON: Closes the 2nd HIL loop: worker completes retrain → notifies agent
          → human approves → agent calls platform promote → investigation done.

Communication between the 3 sub-agents (worker tasks):
  Worker → Platform : HTTP POST (retrain / rollback / replay-test)
  Worker → Agent    : HTTP POST /notify_retrain_complete (retrain only)
  Agent  → Platform : HTTP POST /registry/promote (on 2nd HIL approval)
  All calls are fire-and-forget with retry/DLQ in the worker loop.

Phase 3 test gate (requires Phase 5-6 platform to be working):

  Push manual RETRAIN_SCHEDULED task to Redis →
    Worker logs: task_attempt, task_complete
    Platform: new model version in Staging
    Agent: GET /approvals → 1 pending PROMOTE_TO_PRODUCTION
    Approve it → platform /registry/promote called
    GET /investigations/{id} → status=completed
  Kill Postgres mid-task →
    5 retries → docker exec redis LLEN queue:tasks:dlq → 1
